# API開発ツールガイド — 設計・実装・運用

フロントエンドとバックエンドが分離したSPA + REST API構成のプロジェクトに適用できる汎用ガイド。
技術スタックが変わっても考え方は共通。

---

## 前提：なぜこのガイドが必要か

フロントとバックが分離した構成では以下の問題が必ず起きる。

```
よくある失敗

1. 実装してから仕様を共有する
   → フロントとバックでレスポンスの形が違う

2. ドキュメントを更新し忘れる
   → コードとドキュメントが乖離して誰も信用しなくなる

3. テストを書かない
   → 変更のたびに手動確認が必要になる
   → リグレッション（過去に動いていたものが壊れる）に気づかない
```

このガイドはこの3つを防ぐためのもの。

## 使うツール一覧

### 設計フェーズ
| ツール | 用途 |
|---|---|
| AI（Claude など） | OpenAPIのたたき台を生成する |
| Stoplight | API仕様の設計・チームでの合意・モックサーバー |

### 実装フェーズ
| ツール | 用途 | 対象 |
|---|---|---|
| ドキュメント自動生成ツール | コードからAPIドキュメントを生成 | バックエンド |
| openapi-typescript | OpenAPIから型定義を自動生成 | フロントエンド（TypeScript） |
| ユニットテストツール | 関数・クラス単体のテスト | バックエンド |
| 静的解析ツール | 型エラー・コードの問題を検知 | バックエンド |
| コンポーネントテストツール | UIの動作テスト | フロントエンド |

**技術スタック別の対応ツール**

```
バックエンド
├ Laravel  → L5-Swagger / PHPUnit / PHPStan
├ Rails    → rswag / RSpec / RuboCop
└ Node.js  → swagger-jsdoc / Jest / ESLint

フロントエンド
├ Next.js / React → Vitest + Testing Library
├ Vue             → Vitest + Vue Testing Library
└ 共通            → openapi-typescript（TypeScript共通）
```

### 運用フェーズ
| ツール | 用途 |
|---|---|
| GitHub Actions | CI/CD・テストの自動実行・ドキュメントの自動生成 |
| GitHub Pages | APIドキュメントの公開 |
| Playwright | E2Eテスト（フロントとバックを結合した状態でのテスト） |

## 設計フェーズ

### 原則

**「実装してから仕様を共有する」をやめる。**
コードを1行も書く前にAPIの仕様をフロントとバックで合意する。これだけで仕様ズレの8割は防げる。

### 手順

#### 1. AIでOpenAPIのたたき台を生成する

以下のフォーマットでAIに投げると30秒でYAMLのたたき台が生成される。

```
以下の仕様でOpenAPI 3.0のYAMLを書いて。

エンドポイント：[HTTPメソッド] [パス]
認証：[認証方式]
リクエスト：[フィールド名（型）の一覧]
レスポンス：[フィールド名（型）の一覧]
エラー：[エラーコードと理由]
```

生成されたYAMLはそのままで完璧ではない。たたき台として使ってチームで修正する。

#### 2. StoplightにYAMLを貼ってフロントと合意する

- GUIでエンドポイントの一覧を確認する
- フロントに共有して「このレスポンスで問題ないか」を確認してもらう
- **コードを1行も書く前にここで合意する**

合意のときに必ず確認すること：

```
確認項目

□ どのIDを渡すか（users.id なのか profile.id なのかなど）
□ nullが返る可能性があるフィールドはどれか
□ 配列で返すのかオブジェクトで返すのか
□ ページネーションが必要か
□ 日時のフォーマットは何か（created_at の形式など）
□ 認証が必要なエンドポイントはどれか
```

#### 3. Stoplightのモックサーバーを起動する

- バックの実装を待たずフロントが開発を開始できる状態にする
- フロントはモックサーバーのレスポンスを使って画面を作る
- バックとフロントが並行で開発できる

#### 4. 仕様変更のルールを決める

```
仕様を変えたい
    ↓
まずStoplightのドキュメントを変える
    ↓
フロントに確認する
    ↓
合意してから実装する
```

「実装してから報告する」は禁止。
仕様変更をPRに必ず明記する。

## 実装フェーズ

### 原則

**「コードとドキュメントを同じファイルに置く」**
別ファイルにドキュメントを書くと更新し忘れる。コードのアノテーションとしてドキュメントを書くことでコードを変えたらドキュメントも変える状態にする。

### バックエンド

#### ドキュメント自動生成ツールの使い方

Controllerにアノテーションを書く。コードを変えたら**必ずアノテーションも同時に変える**。

```
アノテーションに書くべき内容

- エンドポイントのパスとHTTPメソッド
- 認証が必要かどうか
- リクエストのフィールドと型（requiredかどうかも）
- レスポンスのフィールドと型
- エラーレスポンス（422・403・404など）
```

#### テストを書く優先度

全部書こうとしない。優先度順に書いていく。

```
優先度1（最初に書く）
認証・認可のテスト
└ ログインしていない状態で弾かれるか
└ 権限のない操作が弾かれるか
└ 他人のデータを操作できないか

優先度2
ビジネスロジックのテスト
└ 重複データが作られないか（冪等性）
└ 重要な計算・条件分岐が正しいか

優先度3
レスポンスのフィールドテスト
└ 見せてはいけないフィールドが返っていないか
└ レスポンスの構造が仕様通りか

優先度4（余裕があれば）
バリデーションのテスト
細かいエッジケースのテスト
```

#### 静的解析ツールの使い方

最初からレベルを上げすぎない。

```
推奨の進め方

最初   低めのレベルで導入してCIに組み込む
↓
慣れたらレベルを上げる
↓
最終的に高いレベルを目指す
```

### フロントエンド

#### openapi-typescriptの使い方

```bash
# OpenAPIのYAMLから型定義を生成するコマンド
npx openapi-typescript [APIドキュメントのパス] -o [出力先のパス]
```

**型は手書きしない。必ずOpenAPIから自動生成する。**
生成した型ファイルは手動で編集しない（次の生成で上書きされる）。

バックがレスポンスを変えたらこのコマンドを再実行する。型エラーが出た箇所を修正する = 仕様のズレが解消される。

#### コンポーネントテストを書く優先度

```
優先度1（最初に書く）
権限によって表示が変わる部分
└ このロールには見えるボタンが別のロールには見えないか

優先度2
重要なフォームの動作
└ 送信できるか
└ バリデーションエラーが出るか

優先度3（余裕があれば）
データの表示
細かいUI状態の変化
```

## 運用フェーズ

### 原則

**「人間の意識に依存しない仕組みを作る」**
「確認しよう」「更新しよう」という意識に頼ると必ず忘れる。CIで機械的に検知する。

### GitHub ActionsのCIで何をチェックするか

```
バックエンド
├ 静的解析（型エラー・コードの問題を検知）
├ ユニットテスト（認証・認可・ビジネスロジック）
├ APIドキュメントの自動生成
├ ドキュメントの差分チェック  ← 差分があればCIを落とす
└ フロントの型再生成 + 型チェック  ← バックの変更でフロントが壊れないか確認

フロントエンド
├ 型チェック（tsc）
├ Lintチェック（ESLint など）
├ コンポーネントテスト（Vitest など）
└ E2Eテスト（Playwright）
```

### GitHub ActionsのCIテンプレート

プロジェクトに合わせて書き換えて使う。

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  # バックエンドのジョブ
  backend:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v3

      # 言語・フレームワークに合わせてセットアップを変える
      # Laravel の場合
      - name: PHPセットアップ
        uses: shivammathur/setup-php@v2
        with:
          php-version: '8.2'
      - name: 依存関係インストール
        run: composer install

      # 静的解析
      - name: 静的解析
        run: [静的解析ツールのコマンド]

      # テスト
      - name: テスト実行
        run: [テストコマンド]

      # APIドキュメントの生成と差分チェック
      - name: APIドキュメント生成
        run: [ドキュメント生成コマンド]
      - name: ドキュメント差分チェック
        run: git diff --exit-code [ドキュメントのパス]

      # バックの変更でフロントの型が壊れていないか確認
      - name: フロントの型を再生成して確認
        run: |
          cd [フロントのディレクトリ]
          npm ci
          npx openapi-typescript [APIドキュメントのパス] -o types/api.ts
          npx tsc --noEmit

      # mainへのマージ時だけドキュメントを公開
      - name: GitHub Pagesにデプロイ
        uses: peaceiris/actions-gh-pages@v3
        if: github.ref == 'refs/heads/main'
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: [ドキュメントのディレクトリ]

  # フロントエンドのジョブ
  frontend:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v3

      - name: Node.jsセットアップ
        uses: actions/setup-node@v3
        with:
          node-version: '20'

      - name: 依存関係インストール
        run: npm ci

      # 型チェック
      - name: 型チェック
        run: npx tsc --noEmit

      # Lint
      - name: Lintチェック
        run: npm run lint

      # コンポーネントテスト
      - name: テスト実行
        run: npm run test

      # E2Eテスト（時間がかかるのでmainへのマージ時だけ走らせる選択肢もある）
      - name: E2Eテスト
        run: npx playwright test
```

### E2Eテストの書き方

**重要な操作だけ書く。全部書こうとしない。**

```
E2Eテストを書くべき操作

□ ログイン・ログアウト
□ ユーザーが最もよく使うメインの操作（購入・応募・投稿など）
□ 権限による表示の切り替え
□ 過去にバグがあった操作
```

E2Eテストのテンプレート：

```typescript
test('[操作の説明]', async ({ page }) => {
    // 1. 前提条件を満たす（ログインなど）
    await page.goto('/login')
    await page.fill('[name=email]', 'test@example.com')
    await page.fill('[name=password]', 'password')
    await page.click('button[type=submit]')

    // 2. テスト対象の操作をする
    await page.goto('/[対象のページ]')
    await page.click('text=[ボタンのテキスト]')

    // 3. 期待する結果を確認する
    await expect(page.locator('text=[表示されるべきテキスト]')).toBeVisible()
})
```

### PRのテンプレート

リポジトリの `.github/PULL_REQUEST_TEMPLATE.md` に置く。

```markdown
## 何をしたか

## 仕様からの変更点
（変更がある場合は必ず書く。関係者への確認済みかどうかも書く）

## チェックリスト
- [ ] APIドキュメントのアノテーションを更新した
- [ ] CIが通っている
- [ ] 仕様変更がある場合は関係者に確認した
- [ ] テストを書いた
- [ ] テストコードを不必要に削除していない
```

## テストコードの管理ルール

```
消してはいけない
├ 認証・認可のテスト
├ ビジネスロジックのテスト
└ 過去にバグがあった箇所のテスト

消していい
├ 対応する機能自体がなくなった
├ 全く同じことをテストしている重複したテスト
└ 何も意味のある確認をしていないテスト
```

テストコードはリグレッション（過去に動いていたものが変更で壊れること）を検知するためのもの。機能が完成しても消さない。

## よくあるミスと対処

### ドキュメントとコードが乖離する
**原因** コードを変えたのにアノテーションを更新し忘れた。
**対処** CIのドキュメント差分チェックでCIが落ちる。アノテーションを更新してから再度pushする。

### フロントの型エラーが突然増える
**原因** バックがレスポンスを変えてAPIドキュメントが更新された。
**対処** `npx openapi-typescript` で型を再生成して型エラーの箇所を修正する。

### バックの変更がフロントに伝わらない
**原因** バックだけpushしてもフロントのCIは走らない。
**対処** バックのCIにフロントの型再生成と型チェックを組み込む。

### 仕様ズレが実装後に発覚する
**原因** 実装前に仕様を合意しなかった。
**対処** 設計フェーズのStoplightで合意してからコードを書き始めるフローを徹底する。

### CIが遅くてPRのレビューが止まる
**原因** E2EテストをすべてのPRで走らせている。
**対処** E2Eテストはmainへのマージ時だけ走らせる。PRではユニットテストとコンポーネントテストだけ走らせる。

### テストが増えるにつれてCIが重くなる
**原因** テストが増えるのは良いことだが、直列で実行すると時間がかかる。
**対処** GitHub Actionsのmatrixを使ってテストを並列実行する。バックとフロントのジョブは最初から並列にしておく。
